TenderShield AI: AI-Based Tender Evaluation Platform

Submission for Theme 3: AI-Based Tender Evaluation and Eligibility Analysis for Government Procurement

1. Understanding the Problem & Realities of Government Procurement

Government procurement in organizations like the CRPF relies heavily on strict compliance, transparency, and fairness. Currently, evaluating complex tender bids is a highly manual, labor-intensive process fraught with risks of human error and inconsistency. Evaluators are forced to cross-reference hundreds of heterogeneous pages (financial statements, MSME certificates, complex technical specs) against dense legal tender criteria.

The core problem is not just "reading documents"—it is trust. The realities of government procurement dictate that an AI system cannot act as a "black box" that silently rejects bidders. Every decision must be auditable, explainable at a microscopic level, and defer to a human officer for edge cases. TenderShield AI is built specifically to address these realities by acting as an "assistant" rather than a "replacement," ensuring fairness, extreme auditability, and human-in-the-loop control.

2. Extracting Eligibility Criteria from Tenders

To extract criteria, the system uses an NLP classification engine that processes the raw tender document. * Separation of Categories: The NLP engine parses the document structure (headings, tables, lists) and uses semantic analysis to classify extracted rules into distinct buckets: Technical (e.g., ISO certifications, product specs), Financial (e.g., turnover thresholds, EMD deposits), and Compliance (e.g., GST registrations, Blacklisting declarations). * Mandatory vs. Optional: We employ rule-based keyword extraction layered with an LLM. Keywords like "must," "shall," "mandatory," and "required" trigger a mandatory flag. Words like "preferred," "desirable," or "optional" flag the criteria as non-binding but scored.

3. Parsing Heterogeneous Bidder Submissions

Private bids arrive in messy, unpredictable formats. Our document processing pipeline utilizes a hybrid approach: * Digital PDFs & Word Files: Standard text extraction (PyPDF2/pdfplumber) is used to cleanly pull raw text. * Scanned Documents & Photographs: An advanced OCR Engine (Optical Character Recognition) coupled with Vision-Language Models (VLMs) is deployed. This allows the system to read stamped certificates, handwritten signatures, and poorly scanned invoices. * Tables: We use specialized layout-aware parsers to extract tabular data, ensuring that row/column associations (like yearly financial turnovers) remain intact. * Data Structuring: All extracted text is converted into structured JSON formats representing key-value pairs (e.g., {"Company_Turnover_2023": "₹6 Crore"}).

4. Matching Criteria & Handling Ambiguity

Once both the Tender Criteria and Bidder Data are structured, the Evaluator module attempts a match. * Matching: The system maps the extracted bidder values to the required criteria (e.g., checking if the parsed GST number matches the required format, or evaluating if Turnover: 6 Crore > Threshold: 5 Crore). * Handling Ambiguity: Legal language and varying formats often create mismatches (e.g., "Rs. Five Cr" vs "₹50,000,000"). The NLP engine is utilized to perform semantic equivalence checks rather than strict string matching. If the engine's confidence score drops below a strict threshold (e.g., 85%), the system refuses to make an automated decision.

5. Explainable Verdicts & Surfacing Ambiguous Cases

A core non-negotiable is the prevention of silent disqualifications. * Criterion-Level Verdicts: For every rule, the system generates a triplet verdict: [Status, Extracted Value, Document Reference]. For example: [Eligible, "₹6 Crore", "Audited_Financials_Page_3.pdf"]. * Need Manual Review: If the OCR cannot read a blurry certificate, or if the LLM detects conflicting information, the status is set to "Need Manual Review". The rationale explicitly states why (e.g., "Financial figures on Page 4 are blurry. Could not verify if turnover exceeds ₹5 Crore"). This surfaces the exact page and problem to the human officer, streamlining their review rather than rejecting the bidder.

6. Guaranteeing Auditability

To meet government standards, TenderShield AI incorporates a persistent database architecture (SQLite/PostgreSQL). * Every uploaded document is hashed (SHA-256) to ensure tamper-proofing. * Every automated decision, confidence score, and extracted rationale is permanently logged in a secure Project Archives relational database. * The final consolidated report includes an immutable audit trail detailing exactly which model version ran the evaluation, at what timestamp, and which human officer (via our Secure Gateway login) signed off on the final verdict.

7. Architecture Overview & Technology Choices

  • Frontend: React.js dashboard providing a premium, responsive UI for officers to manage uploads, view color-coded verdicts, and access the historic Project Archives.
  • Backend: Python (FastAPI/Flask) offering high-throughput API endpoints to handle heavy document processing concurrently.
  • Document AI Pipeline:
    • OCR/Vision: Tesseract / Donut models for robust image-to-text.
    • NLP Engine: State-of-the-art LLMs (e.g., fine-tuned LLaMA-3 or secure enterprise API endpoints like Gemini/GPT-4) for semantic extraction and rationale generation.
  • Storage: Local relational database for the audit trail, ensuring data sovereignty without requiring external cloud storage.

8. Main Risks and Trade-offs

  • Risk: Hallucinations or OCR errors misinterpreting numbers.
    • Mitigation: Implementing a strict "confidence threshold." If the system is not extremely confident, it defaults to "Manual Review."
  • Risk: Data Privacy and Sovereignty. Government documents cannot be sent to public AI clouds.
    • Mitigation: The architecture is designed to support containerized, on-premises deployment utilizing local open-source models if required, ensuring zero data leakage.
  • Trade-off: Processing time vs. Accuracy. Thorough OCR and LLM evaluation takes time. We prioritize extreme accuracy and auditability over instant results, as government evaluations prioritize correctness over sub-second latency.

9. Implementation Plan for Round 2 (Sandbox)

Assuming a sandbox environment with sample documents, our implementation plan is: * Week 1 (Infrastructure): Deploy the React frontend and Python backend into the sandbox. Initialize the secure login gateway and SQLite audit database. * Week 2 (Data Ingestion): Implement the OCR and NLP ingestion pipelines. Process the mock tender documents to extract the baseline criteria schema. * Week 3 (Evaluation Engine): Run the mock bidder documents through the system. Fine-tune the confidence thresholds to ensure perfect explainability and test the "Need Manual Review" flagging logic. * Week 4 (Review & Export): Finalize the human-in-the-loop UI. Ensure officers can easily read the AI's rationale, override decisions if necessary, and export the final, signed PDF report.
